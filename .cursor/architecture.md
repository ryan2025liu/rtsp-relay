# 架构设计规范 (Architecture Guidelines)

## 📋 概述

本文档定义了项目架构设计的基本原则和规范，为 AI 助手和开发者提供架构决策的指导框架。

---

## 🏗️ 架构原则

### 1. 关注点分离 (Separation of Concerns)

将不同的功能职责分离到不同的模块中：

```
┌─────────────────────────────────────────────────────┐
│                  Presentation Layer                  │
│              (UI Components, Views)                  │
├─────────────────────────────────────────────────────┤
│                  Application Layer                   │
│          (Use Cases, Business Logic)                │
├─────────────────────────────────────────────────────┤
│                    Domain Layer                      │
│           (Entities, Business Rules)                │
├─────────────────────────────────────────────────────┤
│                Infrastructure Layer                  │
│      (Database, External Services, APIs)            │
└─────────────────────────────────────────────────────┘
```

### 2. 依赖倒置原则 (Dependency Inversion)

- 高层模块不应依赖低层模块，两者都应依赖抽象
- 抽象不应依赖细节，细节应依赖抽象
- 通过接口/协议定义模块边界

### 3. 单一职责原则 (Single Responsibility)

- 每个模块/类/函数只负责一个功能域
- 当一个模块有多个变更理由时，考虑拆分

### 4. 开闭原则 (Open/Closed)

- 对扩展开放，对修改关闭
- 新功能应通过扩展实现，而非修改现有代码

## 📦 模块设计规范

### 模块边界定义

```
模块应该：
✅ 有清晰的公共 API
✅ 隐藏内部实现细节
✅ 最小化对外部的依赖
✅ 可以独立测试
✅ 有明确的职责范围
```

### 模块通信

```
推荐的通信方式：
1. 直接调用（简单场景）
2. 事件/消息（解耦场景）
3. 依赖注入（可测试性）
4. 接口适配器（外部服务）
```

### 循环依赖处理

**禁止循环依赖！** 如果出现，考虑：
- 提取公共模块
- 使用依赖注入
- 重新设计模块边界
- 使用事件驱动解耦

## 🔌 API 设计规范

### RESTful API 原则

| 操作 | HTTP 方法 | 示例 |
|------|----------|------|
| 获取资源 | GET | `GET /users/123` |
| 创建资源 | POST | `POST /users` |
| 完整更新 | PUT | `PUT /users/123` |
| 部分更新 | PATCH | `PATCH /users/123` |
| 删除资源 | DELETE | `DELETE /users/123` |

### API 命名规范

```
✅ 正确示例：
GET /api/v1/users
GET /api/v1/users/123/orders
POST /api/v1/orders

❌ 错误示例：
GET /api/v1/getUsers
GET /api/v1/user_list
POST /api/v1/createOrder
```

### 响应格式标准

```json
{
  "success": true,
  "data": { ... },
  "message": "操作成功",
  "meta": {
    "page": 1,
    "total": 100
  }
}
```

### 错误响应格式

```json
{
  "success": false,
  "error": {
    "code": "USER_NOT_FOUND",
    "message": "用户不存在",
    "details": { ... }
  }
}
```

## 📊 数据层设计

### 数据库设计原则

1. **规范化**: 消除数据冗余
2. **索引策略**: 基于查询模式优化
3. **约束完整性**: 使用外键和检查约束
4. **命名一致性**: 表名复数，字段名蛇形

### Repository 模式

```
┌──────────────────────┐
│   Business Logic     │
├──────────────────────┤
│   Repository Interface │  ← 抽象层
├──────────────────────┤
│   Repository Impl    │  ← 具体实现
├──────────────────────┤
│   Database/ORM       │
└──────────────────────┘
```

### 缓存策略

| 场景 | 策略 | 适用场景 |
|------|------|---------|
| Cache-Aside | 读时检查缓存 | 读多写少 |
| Write-Through | 写时同步更新 | 一致性要求高 |
| Write-Behind | 写时异步更新 | 高写入吞吐量 |

## 🔄 异步处理设计

### 消息队列使用场景

- 解耦系统组件
- 削峰填谷
- 异步任务处理
- 事件驱动架构

### 异步处理原则

1. **幂等性**: 消息可重复处理
2. **超时处理**: 设置合理超时
3. **重试机制**: 指数退避重试
4. **死信队列**: 处理失败消息

## 🛡️ 安全架构

### 认证流程

```
┌────────┐    1. 认证请求    ┌────────────┐
│ Client │ ───────────────→ │ Auth Server │
└────────┘                   └─────┬──────┘
    │                              │
    │    2. 返回 Token             │
    ← ─────────────────────────────┘
    │
    │    3. 携带 Token 请求资源
    ↓
┌────────────┐
│ API Server │ → 4. 验证 Token
└────────────┘
```

### 授权模型

- **RBAC**: 基于角色的访问控制
- **ABAC**: 基于属性的访问控制
- **最小权限原则**: 只授予必要权限

## 🧪 可测试性设计

### 依赖注入

```
// 推荐：通过构造函数注入依赖
class UserService {
  constructor(
    private userRepository: UserRepository,
    private emailService: EmailService
  ) {}
}

// 避免：硬编码依赖
class UserService {
  private userRepository = new MySQLUserRepository();
}
```

### 接口隔离

- 定义细粒度接口
- 避免"胖接口"
- 每个接口专注单一职责

```markdown
[提议 | 接受 | 废弃 | 替代]

描述问题背景和约束条件

## 理由

## 影响
描述决策的影响

## 替代方案
列出考虑过的其他方案
```

## 🔧 技术选型原则

### 选型考虑因素

1. **团队熟悉度**: 团队是否有经验
2. **社区活跃度**: 是否有良好的社区支持
3. **长期维护性**: 是否有持续维护
4. **性能需求**: 是否满足性能要求
5. **集成成本**: 与现有系统的集成难度

### 避免的陷阱

- ❌ 追求最新技术
- ❌ 过度设计
- ❌ 忽视运维成本
- ❌ 单点故障风险

## 📝 AI 助手指南

当 AI 助手需要进行架构相关的工作时：

1. **先理解现有架构**: 查看架构文档和代码结构
2. **遵循已有模式**: 使用项目中已建立的模式
3. **小步修改**: 避免大规模重构
4. **讨论大变更**: 重大架构变更前与用户讨论
5. **记录决策**: 重要决策应该记录理由
